# 整体系统总览

这份文档用于描述业务系统级认知，而不是某个单独代码仓库的说明。它的目标是先建立系统边界、服务分工和关键链路视角，再往下承接各仓库、各服务组、各流程和各模块文档。

## 文档定位

这份文档回答的是：

- 整个系统是做什么的
- 系统由哪些服务组成
- 各服务大致负责什么
- 服务之间如何协作
- `market-server` 在整个系统中的位置是什么

## 一、系统整体分层

当前系统可以按责任边界分成三个部分：

1. 上游数据商
2. 我们开发和维护的业务系统
3. C 端 app（95e）

其中：

- 当前文档主要关注的是中间这一层，即我们开发和维护的业务系统，不关心数据商以及95e的开发

## 二、系统目标

这套业务系统的核心目标，是接收上游数据商推送的赛事与盘口数据，完成预处理、聚合、操盘、推送，并为前端、订单、结算和 C 端用户提供稳定一致的数据输出。

### 当前可确认的核心能力

- 接收上游赛事与盘口数据
- 对数据进行预处理和纠错
- 对多来源数据进行聚合和完整性校验
- 将赛事信息和盘口信息按职责分发到不同服务组
- 将处理后的数据继续推送给前端、订单、结算和 95e

## 三、系统边界

### 系统内
当前主要关注的是我们开发和维护的业务系统，已知包含：

- 水机
- 聚合
- 操盘
- 推送
- 订单
- 结算
- 前端

### 系统外
不属于当前团队负责范围的部分：

- 上游数据商
- C 端 app（95e）

## 四、服务全景

当前已知的系统角色与职责如下：

| 角色/服务 | 类型 | 主要职责 | 上游 | 下游 | 备注 |
|---|---|---|---|---|---|
| 上游数据商 | 外部系统 | 推送赛事信息和盘口信息 | 外部数据源 | 水机 | 不由当前团队负责 |
| 水机 | 内部系统 | 整合上游数据并做预处理，过滤明显错误数据 | 上游数据商 | 聚合 | 数据进入内部系统后的第一层处理 |
| 聚合 | 内部系统 | 基于水机数据进一步处理错误信息，保证数据准确性和完整性 | 水机 | 推送组、操盘组 | 通过 Kafka 向下游分发 |
| 操盘组 | 内部系统/服务组 | 负责赛事级别信息，如比分、走秒等 | 聚合 | 操盘前端等 | 更关注赛事级信息 |
| 推送组 | 内部系统/服务组 | 负责盘口信息，如盘口状态、调水、跳水等，并向下游分发 | 聚合 | 操盘前端、95e、订单、结算等 | 当前主要关注范围 |
| 订单 | 内部系统 | 获取相关数据支撑订单业务 | 推送组等 | 待补充 | 当前仅明确会消费相关数据 |
| 结算 | 内部系统 | 获取相关数据支撑结算业务 | 推送组等 | 待补充 | 当前仅明确会消费相关数据 |
| 前端 | 内部系统 | 面向内部操盘展示 | 推送组、操盘组 | 内部用户 | 当前已知存在操盘前端 |
| 95e | 外部系统 | 面向 C 端用户展示和交互 | go-stomp / c-server 等 | C 端用户 | 不由当前团队负责 |

## 五、已知工作职责主链路

当前已知的一条系统主职责链路如下：

0. 上游数据商向内部系统推送赛事信息和盘口信息。
1. 水机先对数据进行整合和预处理，例如聚合多个数据商的数据，过滤掉部分错误数据。
2. 水机将处理后的实时数据通过 Kafka 推送到聚合。
3. 聚合基于水机数据继续处理错误信息，尽量保证数据的准确性与完整性。
4. 聚合再通过 Kafka 将数据推送到推送组和操盘组。
5. 操盘组与推送组分别处理不同的数据职责：
   - 操盘组主要负责赛事级别信息，如比分、走秒等。
   - 推送组主要负责盘口级别信息，如盘口状态、调水、跳水等。
6. 推送组将结果继续推送给操盘前端、95e 页面，同时订单和结算也会获取对应数据。

## 六、当前关注的推送组服务

当前所在的是推送组，现阶段优先关注以下服务：

### 1. market-server
包含两个服务形态：

- `market-server` server 服务
  - 对外提供 API 接口
  - 通过 Redis PubSub 方式发送事件通知给 consumer

- `market-consumer` consumer 服务
  - 处理上游 Kafka 数据
  - 处理 Redis PubSub 事件数据
  - 对处理后的结果继续向下游 Kafka 推送，或者通过 PubSub 推送数据

### 2. c-push-server
职责上更像推送汇聚层。

当前理解：

- 各组都会向 `c-push-server` 推送数据
- 它会对数据做进一步整理
- 然后继续向下游推送 Kafka 数据

### 3. go-stomp
职责相对单一明确：

- 消费 `c-push-server` 推送的 Kafka 数据
- 通过 WebSocket 和 API 两种方式向 95e 推送数据
- 当前已知它会通过 stomp 协议向用户推送数据

### 4. c-server
当前认知还不充分。

当前推测：

- 更偏向服务 95e
- 可能维护部分用户信息与用户配置

## 七、关键数据流与消息流

### 系统级主数据流
- 上游数据商产生赛事和盘口数据
- 水机接收并预处理
- 聚合进一步纠错和补全
- 聚合通过 Kafka 分发到推送组和操盘组
- 推送组 / 操盘组分别向不同下游输出数据

### 一条已确认的真实链路

#### LOL 电竞盘口链路
- 触发源：聚合
- 上游 Topic：`nw.aggregatorServer.mergeMarket.esport-lol.topic`
- 第一步消费：`market-consumer` 消费该消息
- 中间输出：
  - 向 Redis PubSub 推送数据
  - 向 Kafka Topic `nw.marketServer.oddsDiffEsportLOL` 推送数据
- 下游 1：`push-server-gw` 接收后向操盘前端推送数据
- 下游 2：`c-push-server` 接收该 Kafka 数据后，继续向 Kafka 推送统一 Topic `nw.cPushServer.oddsDiffMarket`
- 下游 3：`go-stomp` 消费 `nw.cPushServer.oddsDiffMarket`
- 最终输出：`go-stomp` 通过 stomp 协议向用户推送数据

## 八、market-server 在系统中的位置

`market-server` 属于推送组关注范围内的核心服务组之一，不是整个系统本体。

当前已知定位：

- 它位于聚合之后、C 端推送链路之前
- 它同时承接 API 请求和异步数据处理两类职责
- 它处理的数据重点偏向盘口相关信息
- 它既可能向 Redis PubSub 发事件，也可能向 Kafka 向下游发数据
