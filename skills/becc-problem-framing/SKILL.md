# becc-problem-framing

用于在进入搜索、范围分析、设计或编码前，先把问题本身收拢到可行动状态。

## 何时使用

以下场景优先使用这个 skill：
- 用户只给了一个模糊目标
- 任务目标看起来简单，但成功标准不清楚
- 用户同时提到了问题、限制、偏好，但边界未整理
- 还无法判断接下来应先查证据、先做 scope，还是先做设计
- 如果直接编码，大概率会返工

不要把这个 skill 用于：
- 已经边界清楚、只差执行的纯机械任务
- 已经完成 framing、当前只差验证或 debug 的任务

## 核心意图

这个 skill 的目标不是给方案，而是先回答四个问题：

1. 真实要解决的问题是什么
2. 当前已知约束是什么
3. 当前未知但关键的点是什么
4. 什么结果才算完成

如果这四点不清楚，不应直接进入后续实现。

## 所在 loop

这个 skill 主要位于：
- **Problem Convergence Loop**

典型循环：
- 输入目标
- 复述问题
- 收敛约束
- 标出未知点
- 判断是否需要补证据
- 进入下一步，或继续收敛

## 不可妥协的规则

在进入搜索、设计或编码前，助手必须：
1. 用一句话复述当前任务
2. 明确当前约束，而不是隐含假设
3. 区分已知事实与未确认推断
4. 明确成功标准或验收标准
5. 若关键未知点过多，先停下来要求补信息或补证据

助手不得：
- 把模糊目标直接包装成确定需求
- 把用户偏好当成已经确认的物理约束
- 在成功标准不清时直接进入实现
- 把猜测写成事实

## 输入

输入通常来自：
- 用户自然语言任务描述
- 当前代码仓库上下文
- 已知业务限制
- 已知技术约束
- 已有证据或日志摘要

## 输出

输出至少应包含以下部分：

### 1. Problem Statement
- 当前要解决什么问题
- 目标对象是什么
- 任务的最小边界是什么

### 2. Known Constraints
- 已知技术约束
- 已知业务约束
- 已知环境或流程约束

### 3. Unknowns
- 当前仍缺哪些关键信息
- 哪些未知点会直接影响后续路径

### 4. Success Criteria
- 什么结果算完成
- 至少需要哪些验证或交付物

### 5. Recommended Next Step
只允许给出以下几类下一步之一：
- 补证据
- 做 impact discovery
- 做 design
- 直接执行简单任务

## 停下来询问的条件

遇到以下情况必须停下：
- 无法清楚复述任务目标
- 关键约束互相冲突
- 成功标准不存在或明显缺失
- 已知信息不足以判断下一步该走哪条 loop

## 完成态要求

只有在以下条件成立时，才可把 framing 视为完成：
- 问题陈述清楚
- 约束已列出
- 未知点已显式列出
- 成功标准已明确
- 下一步路径可判断

否则结果必须标记为：
- `partial`：已有部分 framing，但未知点仍高
- `blocked`：缺少用户输入或关键证据

## 推荐输出语气

应保持：
- 短
- 硬约束导向
- 不做宽泛发挥
- 不提前进入实现细节

## 与其他资产的关系

这个 skill 之后通常连接到：
- `becc-evidence-capture`
- `becc-impact-discovery`
- `becc-impact-scope`
- `becc-verification-gate`

它的职责是：

> 先把问题变得可行动，再决定走哪条后续路径。
