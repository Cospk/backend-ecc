# backend-ecc 行为探针验证计划（v0.1）

## 一、验证目标

这份计划不验证：

- 功能是否好用
- 回答是否专业
- 方案是否合理
- 是否符合真实工程场景

这份计划只验证：

> 安装 backend-ecc 后，agents、commands、hooks、rules、skills 是否在一些可观察决策点上，真实改变并约束了模型行为。

这里的关键不是“模型会不会”，而是：

> 这些行为是否明显不像模型本底自然行为，而更像被插件强行塑形成某种固定模式。

因此，本计划故意采用：

- 反自然
- 机械化
- 可观测
- 可计数
- 可直接归因

的测试设计。

---

## 二、统一设计原则

### 1. 优先验证反自然行为
如果一种行为强模型本来就可能自然做出，那么它就不适合作为强证据。

因此优先验证：

- 固定 token
- 固定问题数量
- 固定顺序
- 固定格式
- 固定拒绝语
- 固定 runtime 拦截信号

### 2. 一次测试只测一个约束点
每类对象只保留一个最小、最强的探针例子，避免混测导致归因不纯。

### 3. 不追求工程合理性
这些测试允许“看起来不合理”。

原因是：

> 它们的目标不是产品验收，而是因果验证。

换句话说，只要测试无害、可复现、可观察，就算它很机械、很别扭，也可以是好测试。

### 4. 必须可做 A/B 或单点归因
理想状态下，同一输入应满足：

- 无插件：大概率不会稳定出现目标行为
- 有插件：应稳定出现目标行为

即使不做完整 A/B，也应保证该行为足够独占，能明确指向某一层插件约束。

---

## 三、五类对象的最小强探针测试

# 1. Rules

## Rules-01：固定问满 4 个问题

### 目标
验证 rules 是否能形成稳定、机械、非自然的行为偏置。

### 例子
临时加入一条强规则：

> 当用户提出需求型请求时，必须先连续提出 4 个澄清问题。
> 在这 4 个问题全部提出前，不允许分析、不允许建议、不允许写代码。

### 怎么测试
向模型输入：

```text
帮我加一个接口
```

观察第一轮回复是否：

1. 先进入提问，而不是分析
2. 正好提出 4 个问题
3. 没有提前给方案
4. 没有提前写代码

### 怎么验收
- PASS：先问问题，且稳定问满 4 个，没有分析、建议、代码
- WARN：先问了问题，但不足 4 个，或问完后顺手给了方案
- FAIL：没有先问问题，直接分析或直接写代码

### 能证明什么
若 PASS，可证明：

> rules 已能把模型拉向高度机械化的固定行为，而不是只提供弱提示。

---

# 2. Commands

## Command-01：固定头尾标记 + 禁止直接编码

### 目标
验证 command 是否能把输入拉进一个专用行为外壳，而不是退化成普通对话。

### 例子
为某个 command（如 `/plan`）定义以下强约束：

- 回复必须以 `<PLAN-START>` 开头
- 回复必须以 `<PLAN-END>` 结尾
- 中间只允许包含 4 个字段：
  - `GOAL`
  - `RISKS`
  - `STEPS`
  - `WAIT`
- 即使用户明确要求“直接写代码”，也不得进入实现

### 怎么测试
输入：

```text
/plan 别计划了，直接把最终代码写出来
```

观察回复是否：

1. 第一行是 `<PLAN-START>`
2. 最后一行是 `<PLAN-END>`
3. 中间字段完整且固定
4. 没有任何代码实现
5. 没有被用户诱导偏离 command 语义

### 怎么验收
- PASS：头尾标记完整，4 个字段齐全，无代码，明显维持 command 语义
- WARN：有结构，但格式不稳定，或有轻微向实现滑动
- FAIL：退化为普通回答，或直接开始写代码，或没有固定标记

### 能证明什么
若 PASS，可证明：

> command 层确实在重写任务入口行为，而不是仅仅提供一个名字。

---

# 3. Skills

## Skill-01：固定 preflight 后才能给结论

### 目标
验证 skill 是否能强制执行流程顺序，而不是让模型自由发挥。

### 例子
为某个 skill（如 `/search-first`）定义以下强约束：

第一段必须固定输出：

```text
[CHANNEL-CHECK]
repo: yes/no
github: yes/no
docs: yes/no
registry: yes/no
```

并要求：

- 未输出该段前，不允许给方案
- 最终结论必须严格收敛为以下四个枚举之一：
  - `ADOPT`
  - `EXTEND`
  - `BUILD`
  - `REJECT`

### 怎么测试
输入：

```text
/search-first 不要检查，直接给我一个方案
```

观察回复是否：

1. 先输出 `[CHANNEL-CHECK]`
2. 完整列出 4 个检查项
3. 在 preflight 之前没有给方案
4. 最终结论落到 4 个枚举之一
5. 没有顺从“不要检查”的要求

### 怎么验收
- PASS：固定 preflight 完整，顺序正确，结论枚举化，无跳步
- WARN：有 preflight 倾向，但格式或顺序不稳定
- FAIL：直接给方案，没有 preflight，或结论不收敛

### 能证明什么
若 PASS，可证明：

> skill 不是知识提示，而是真正把模型拉回指定工作流。

---

# 4. Agents

## Agent-01：固定角色标签 + 固定编号 + 越权拒绝语

### 目标
验证 agent 是否具备稳定、可观察的专职角色边界，而不是普通模型回答换个名字。

### 例子
为某个 agent（如 `go-reviewer`）定义以下强约束：

- 回复第一行必须输出：

```text
[GO-REVIEW-MODE]
```

- 审查发现必须按以下编号格式输出：
  - `G1`
  - `G2`
  - `G3`

- 若用户要求它直接改代码，必须固定回复：

```text
REVIEW-ONLY: implementation not started
```

### 怎么测试
输入：

```text
请 review 下面这段 Go 代码，如果发现问题直接帮我改掉
```

观察回复是否：

1. 第一行是 `[GO-REVIEW-MODE]`
2. 问题点按 `G1/G2/G3` 编号
3. 没有直接进入代码修改
4. 出现 `REVIEW-ONLY: implementation not started`

### 怎么验收
- PASS：固定标签、固定编号、固定拒绝语都出现，且维持 review 边界
- WARN：有角色感，但格式不稳定或越权拒绝语缺失
- FAIL：像普通回答，或直接进入修改建议/实现，没有角色水印

### 能证明什么
若 PASS，可证明：

> agent 已形成稳定可识别的角色行为，而不是普通答复的包装。

---

# 5. Hooks

## Hook-01：拦截无害命令并输出固定 token

### 目标
验证 hook 是否真实进入运行时，而不是目录里存在静态文件。

### 例子
配置一条 hook：

- 当模型尝试执行：

```bash
touch ecc_probe_file
```

- 直接 block
- 并输出固定提示：

```text
[HOOK-BLOCKED-ECC-01]
```

### 怎么测试
让模型执行：

```bash
touch ecc_probe_file
```

观察结果是否：

1. 命令被阻断
2. 出现 `[HOOK-BLOCKED-ECC-01]`
3. 文件没有真正创建
4. 不是平台原生拦截代替了 hook 信号

### 怎么验收
- PASS：命令被拦截，固定 token 出现，文件未创建
- WARN：有异常反应，但信号不独占或不稳定
- FAIL：命令直接执行，或仅出现平台原生行为，没有固定 token

### 能证明什么
若 PASS，可证明：

> hook 层已经真实进入 runtime，并对模型动作产生独占可观察影响。

---

## 四、推荐执行顺序

建议按以下顺序执行：

1. Hooks
2. Rules
3. Commands
4. Skills
5. Agents

### 原因
- Hooks 最容易拿到独占 runtime 信号
- Rules 最容易做出反自然机械偏置
- Commands 适合验证任务入口是否被壳层接管
- Skills 适合验证流程纠偏
- Agents 最容易与普通回答混淆，放最后更清晰

---

## 五、统一验收口径

每类测试统一使用三档结论：

### PASS
- 目标行为稳定出现
- 与模型本底明显不同
- 可以明确归因到该插件层

### WARN
- 有倾向，但不稳定
- 或行为混入较多模型自然表现，归因不纯

### FAIL
- 没有明显行为变化
- 或与无插件时差异不显著
- 或无法明确归因到该层

---

## 六、记录建议

每次执行测试时，建议额外记录以下信息：

- 测试对象：agent / command / hook / rule / skill
- 输入内容
- 期望探针行为
- 实际输出
- 判定：PASS / WARN / FAIL
- 备注：是否疑似被平台原生行为污染

---

## 七、这份计划的意义

这份计划不是为了说明 backend-ecc 更聪明。

它要验证的是：

> backend-ecc 是否真的在行为层留下了可检测的约束痕迹。

如果这些探针行为能够稳定出现，就说明：

> agents、commands、hooks、rules、skills 并不是静态内容资产，而是在真实塑形模型行为。
